Why we built it this way
Cluster · the canonical scene

Layout, not behaviour

cn-platform owns the apex-and-ring layout, nothing more. No hover-reveal, no side lists, no connector lines pinning apps to specific workspace sides. Those richer interactions live in bespoke surface components (e.g. <platform-diagram> on the homepage) that compose on top.

  • CSS Grid for placement. A 5×3 grid with explicit ::slotted(:nth-child) rules pins the six neighbours into a hex ring around the apex. No JS positioning, no math at runtime.
  • Up to six apps, no enforcement. Pass three for a partial ring, six for a full one. The grid simply leaves empty cells where no app is provided.
  • Apex and apps are slot content. Use cn-hex-prism for the standard ConNext look, or cn-hex for a flatter alternative. Mix them if a page needs both.
  • Ground is opt-in. The platform shadow is suggestive, not load-bearing. Off by default; turn it on for hero scenes.
  • Tone abstracts background. Three tones cover every page context (light kit page, dark hero, embedded inside a card).
  • Responsive falls back to a 2-column stack. Below 700px the hex ring becomes a vertical 2-col grid, apex on top.
  • "Kernel" is a banned word. The central element is the workspace, not the kernel. Slot is named apex; copy and kicker labels say workspace. See identity / voice.
Uses cn-hex-prism as the canonical apex/app element. Works equally with cn-hex. Used by Future homepage hero, product overview pages, ecosystem visualisations.
