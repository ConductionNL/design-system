AppMock — abstract product previews.

Token-built abstract panels of each Conduction app. Used wherever a marketing surface needs to show an app without embedding a real screenshot. Reference for the level of abstraction: honeycomb.io/technologies/* — recognisable as the product, never literal.

Anatomy

One chassis. Two patterns.

Every AppMock variant is composed from the same atom toolkit. A topbar across the top is always there; below it the body is either an App (left nav + main work surface, optionally a right detail rail) or a Desk (no nav, the topbar's Nextcloud app shelf is the navigation). Both patterns share the same 16:10 frame, the same tokens, the same orange-accent budget.

App pattern

topbar · nav · col [· detail]

Default for any Conduction app rendered inside Nextcloud's app surface. Carries its own per-app navigation in the left rail because the topbar's Nextcloud shelf points outward (other Nextcloud apps), not inward (this app's screens).

Used by: OpenRegister, OpenCatalogi, OpenConnector, ZaakAfhandelApp, Procest, DeciDesk, DocuDesk, OpenWoo, NLDesign, LarpingApp, PipelinQ, SoftwareCatalog.

Desk pattern

topbar · col (full-bleed)

Personal home / dashboard view. The Nextcloud topbar shelf is the only navigation; the body is one full-width canvas of widgets. Used when the surface aggregates across the whole workspace instead of belonging to one app.

Used by: MyDash, MyDash · Tiles, MyDash · BI, MyDash · Widgets.

Zones

Four atoms, one chassis.

Each card focuses on one zone of the chassis. The zone is at full opacity; the rest of the frame is dimmed so you see where the zone sits, not just what it is. The orange outline marks the focused region.

.col

Main column

App · Desk · always

The work surface. In an App pattern, this is one column with stacked content (KPI strip on top, panels or tables below). In a Desk pattern, .col nests inside .grid to become a 4-column widget canvas, full-bleed across the body.

Inside
App pattern: .kpiRow + .panelRows + tables. Desk pattern: .grid with .w widgets
Width
flex: 1; min-width: 0; overflow: hidden
Tokens
White bg; tokens scope to inner panels (--c-cobalt-100 borders, --c-mint-500 / --c-orange-knvb KPI accents)

Detail rail

App · optional · Desk: never

The right-hand rail. Optional. Use it when the variant genuinely surfaces a selected record, a filter summary, or pinned metadata. If the rail would be empty or padded with placeholders, drop it.

Inside
Stacked .rows with modifiers: .head (label), .dark (emphasised value), .accent (orange highlight), .short (half-width)
Width
26%, min 130px; flex-shrink: 0
Tokens
White bg; --c-cobalt-100 left border; --c-orange-knvb for at most one .row.accent

  • One .row.accent max per rail; use it for the row the user came in to see
  • Don't fill it with chrome (nav, breadcrumb, tabs); those go in .col or .topbar
Authoring rules

Six constraints, every variant.

Tokens only, no images.

Every fill, stroke, and text colour is a CSS variable from tokens.css. No SVG icons, no PNGs, no real screenshots. Heights and widths render placeholder lines.

One orange accent per variant.

KNVB orange (--c-orange-knvb) is reserved for the single most-emphasised element in the mock: a primary action, a peak bar in a chart, the active hex in the nav header. Two oranges in one frame is a bug.

Status palette, applied to glyphs.

Mint (--c-mint-500) signals stable / done. KNVB orange signals beta / accent. Cobalt-blue is neutral. Avoid raw red except in genuine error states.

16:10 frame, three sizes.

The outer .frame is always aspect-ratio: 16 / 10. Three width caps: sm 480 px, md 720 px (default), lg 960 px. Don't override the aspect ratio.

No real text.

Lines, headings, labels, and stats are rendered as solid-colour rectangles. The viewer should recognise the shape of the screen, not read its content. Real text breaks the abstraction.

Recognisable, not literal.

Each variant pictures the canonical view of one Conduction app: the screen a user would land on first. Take a real screenshot at localhost:8080, count the regions, then redraw with the atom toolkit.

Status. Three of six are based on real screenshots taken at localhost:8080 (MyDash, OpenRegister, DeciDesk). Once those builds finish and the screenshots are in, the three INFERRED variants will get a refinement pass.
