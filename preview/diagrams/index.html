Design system  /  Diagrams

Diagrams.

Brand-strict diagram primitives as web components. Every shape is a pointy-top hex, every colour comes from the palette, every animation respects the brand. Drop the module into any HTML page; works in plain HTML, Vue, React, Docusaurus, Nextcloud apps. Package readme.

Brand rules · the non-negotiables

Pointy-top only

Hexes never rotate. The point is at the top, always. Flat-top hexes belong to other brands.

Solid colours only

No gradients on the shape itself. No rgba alpha. Backgrounds can be hex polygons; the shape stays one colour.

One orange per scene

KNVB orange is the highlight. Use it once per diagram, on the node you want the eye to land on. Never as a primary fill.

Fixed palette

cobalt, orange, mint, lavender, terracotta, forest, nextcloud-blue, red. No off-brand colours. Components reject anything else.

Hexes indicate, rectangles do

Hex prisms are "the system". Rectangles (side-boxes) are what feeds it or consumes it. The shape carries hierarchy.

Composition over configuration

Diagrams are composed from <cn-*> elements. Each element renders one shape. The diagram is the layout of those shapes.

Walk-through · build a domain map in four steps

A worked example. We will build the same diagram that lives on identity / domains: a brand apex with two externally-linked product surfaces. Every step shows the visual on the left and the code on the right. You will write only HTML, no JavaScript.

Step 01

Place the apex hex

Start with the brand. conduction.nl is the root of everything; render it as a large cobalt hex with the brand-mark slot. Cobalt is the default colour, so no color attribute is needed.

The label is plain text in the default slot.

conduction.nl 
<!-- Step 1: the apex -->
<cn-hex size="xl">conduction.nl</cn-hex>
Step 02

Add the product surfaces

Two products link out from the apex: connext.conduction.nl (the platform proposition) and apps.conduction.nl (the app store). Render them as medium hexes flanking the apex.

We are still using the default cobalt; the diagram reads as one family.

connext conduction.nl apps 
<!-- Step 2: flanking products -->
<cn-hex size="md">connext</cn-hex>
<cn-hex size="xl">conduction.nl</cn-hex>
<cn-hex size="md">apps</cn-hex>
Step 03

Land the orange accent

One node should be the eye-catcher. The app store is what we want visitors to click, so we promote it with color="orange". This is the one orange-per-scene rule, in practice.

Notice that we never write a hex code; the colour name is the API.

connext conduction.nl apps 
<!-- Step 3: one orange accent -->
<cn-hex size="md">connext</cn-hex>
<cn-hex size="xl">conduction.nl</cn-hex>
<cn-hex size="md" color="orange">apps</cn-hex>
Step 04

Icons for clarity, interactive for clickability

Add an icon SVG to the icon slot to make each surface scannable at a glance. Add interactive on nodes that link out, you get hover scale, focus ring, and Enter/Space activation.

The icon and label co-exist; default layout stacks them vertically.

connext conduction.nl apps 
<!-- Step 4: icons + interactive on the linking nodes -->
<cn-hex size="md" interactive>
  <svg slot="icon" viewBox="0 0 24 24" ...>...</svg>
  connext
</cn-hex>

<cn-hex size="xl">
  <svg slot="icon" ...>...</svg>
  conduction.nl
</cn-hex>

<cn-hex size="md" color="orange" interactive>
  <svg slot="icon" ...>...</svg>
  apps
</cn-hex>

That is the whole technique. Lay out shapes, pick colours by name, add slots for icons, mark the clickable ones interactive. The connections between hexes (lines, arrows, branches) come from the next-up components: <cn-hex-prism> for system nodes, <cn-domain-tree> for the layout itself. Until those land, you compose by hand with flexbox and SVG lines.

Why we built it this way

One render path, many surfaces

Diagrams show up everywhere: the kit, the website, product docs, the CMS, internal apps. They all need to look identical. Building once and shipping as web components is the only way to keep that promise.

  • No third-party diagram library. Mermaid, drawio, React Flow all force compromises against the brand (rounded boxes, fixed palettes, gradient drop-shadows). We render shapes ourselves so every hex is a hex.
  • Web components, not framework components. Plain HTML pages can use them. Vue 2 apps (nextcloud-vue) can use them. React (Docusaurus) can use them. Future Lit-based work can use them. One implementation, every surface.
  • No build step until we publish. Drop a <script type="module"> tag and start composing. The kit ships today, npm later.
  • Composition over configuration. A diagram is HTML. You write hexes, you arrange them. No DSL, no JSON schema, no canvas. The browser is the renderer.
  • Animation is opt-in. CSS handles hover and transitions. Web Animations API handles keyframes when needed. Motion One ships with cn-platform when we orchestrate sequences. The base hex animates nothing it does not need to.
